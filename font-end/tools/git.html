一、对比Git和SVN ​

  • SVN：集中式版本控制系统：将项目历史版本集中到一台服务器管理，各个客户端通过这台服务器工作（拉取是最新的项目版本）
    • 优点：由专门的管理员去管理项目版本
    • 缺点：这台服务器不能宕机，不能有故障
  • Git：分布式版本控制系统：将项目历史版本分布到各个客户端（客户端拉取是有历史记录的完整的项目仓库），可以使用远程仓库进行团队协作。
    • 优点：回滚速度快，占空间不大

二、Git用法 ​

分区 ​

  • 工作区：编辑项目源码的区域
  • 暂存区：把我们想要存储的内容放在暂存区
    • 查看本地缓存区状态 git status
  • 版本库：把暂存区里面的内容拿出来作为一个的历史版本保存
    • 查看本地仓库历史区信息 git log

配置用户名和电子邮件（团队协作时需要） ​

每次Git提交时都会引用，说明是谁提交了更新，会随更新内容一起被永久纳入历史记录

js
// 当前用户下的配置文件：C:\Users\HASEE\ .gitconfig
// 查看配置
git config --global --list
// 配置用户名
git config --global user.name 'hexueyi'
// 配置电子邮件
git config --global user.email 'hexueyi126@163.com'

不同优先级别的配置文件：

js
// {git安装目录}\etc\gitconfig 所有用户的配置文件，对当前电脑的所有用户都有效，最低优先级别
git config --system ...
// C:\Users\HASEE\.gitconfig 当前用户的配置文件，对当前用户HASEE有效，中等优先级别
git config --global ...
// 当前项目/.git/config 当前项目的配置文件，只对当前项目有效，最高优先级别
git config ...

1. 初始化git仓库 ​

git init

  • .git目录介绍

    |-hooks 目录包含客户端或服务端的钩子脚本 |-info 包含一个全局性排除文件 |-logs 保存日志信息 |-objects 目录存储所有数据内容【重要】 |-refs 目录存储指向数据的提交对象的指针（分支）【重要】 |-config 文件包含项目特有的配置选项 |-description 用来显示对仓库的描述信息 |-HEAD 文件指示目前被检出的分支（分支）【重要】 |-index 文件保存暂存区信息【重要】

2. 将工作区的修改添加到暂存区 ​

工作区的修改：文件的新增和删除，文件内容的修改，文件名的修改

  • git add 文件名.尾缀名 添加单个文件
  • git add 文件夹/ 添加单个文件夹
  • git add . 添加项目里面所有文件和文件夹
  • 注意：
    • git在将工作区添加到缓存区对于空文件夹无法添加，只有空文件夹下面有文件的时候才能添加
    • 所以此时一般可以有一个特殊文件 .gitkeep文件可以当项目文件夹下面还不知道要写什么东西时候使用
    • 忽略某些不需要生成版本文件和文件夹.gitignore

附：取消暂存 ​

  • 第一种语法 git reset HEAD -- 文件名.尾缀名 取消暂存一个文件 git reset HEAD -- 文件夹/ 取消暂存一个文件夹 git reset HEAD -- . 取消暂存所有

  • 第二种语法
    git rm --cache 文件名.尾缀名 取消暂存一个文件 git rm -r --cache 文件夹/ 取消暂存一个文件夹 git rm -r --cache . 取消暂存所有

3. 将暂存区提交到版本库 ​

git commit -m '本次版本的提交说明'

附：版本库穿梭（历史版本穿梭） ​

  • git reset --hard HEAD^ 回退前一个版本
  • git reset --hard 历史版本号 穿梭到指定版本

4. 查看状态 ​

js
git status // 用于检查工作区状态
git diff   // 查看修改的没有被暂存的文件的对比
git diff --staged   // 查看被暂存的没有被提交的文件的对比
git ls-files -s   // 查看暂存区
git log // 查看版本库
git log --oneline // 查看版本库

5. git分支（平行世界） ​

分支 ​

HEAD是一个指针，默认指向master分支，切换哪个分支就是让HEAD指向哪个分支，当有新提交对象时，HEAD会带着当前分支一起指向这个新提交对象

git 分支规范 ​

  • master： 主分支，只保存可以上线的项目版本（绝对不能在 master 上直接提交代码）
  • develop： 主开发分支，由 master 开辟（绝对不能上传代码，只接受合并）
  • feature-xxx: 主开发功能分支，由 develop 开辟，开发完毕以后，合并到 develop 分支
  • feature-xxx-fix: 是指某一个分支出现的 bug 进行修复，修复完毕以后合并到出现 bug 的功能分支上
  • hotfix： 紧急bug，是指已经上线的项目出现了临时由重大的错误需要马上修改，从 master 分支上开辟一个分支进行临时修复，修复完毕以后直接合并到 master 分支

跟踪分支 ​

git branch  // 查看所有的分支
git log --oneline   // 查看当前分支图
git log --oneline --decorate --graph --all // 查看完整分支图
git log  // 查看提交作者，提交日期，提交说明

创建分支 ​

我们可以直接在远程仓库里面新建分支，一般不推荐，一般都是在本地创建好分支之后推送到远程。明确基于哪个分支创建一个新分支

git branch 分支名

附：推送远程某仓库某分支 ​

git push 远程仓库名 远程分支名

切换分支 ​

切换分支会动三个地方：HEAD；暂存区；工作目录。每次切换分支前，保证当前分支已提交（干净的）。

git checkout 分支名
git checkout -b 分支名    // 新建一个分支并且切换过去

删除分支 ​

不能在本分支删除本分支，切换到其他分支才能删除本分支。删除的是本地分支。不要删除远程分支。

git branch -D 分支名

新建一个分支，并指定指向一个提交对象 ​

比如在主分支上执行，可以回溯历史版本到新分支，即版本穿梭

git branch 分支名 版本号

合并分支 ​

如将test分支合并到master分支，一定要切回master分支

git merge 分支名

远程仓库协同开发 ​

注意协同开发顺序：代码写完，1.先本地提交，2.然后pull同步远程仓库，3.最后push推送

  1. 创建远程仓库 (注：创建一个空的仓库)
  • 远程仓库包括私有项目和开源（公开）项目
  • 私有项目只能自己或者自己添加的人员下载项目代码和提交项目代码
  • 开源项目任何人都能查看和下载项目代码，但是不能提交项目代码

  1. 管理远程仓库地址

    项目地址分为两种: HTTPS 推送的时候需要输入你远程仓库的用户名和密码（你注册的用户名和密码） SSH 需要在你本地生成SSH公钥，然后在远程仓库添加公钥，这样就可以上传 （生成公钥比较麻烦，时间比较长，一般项目开发推荐第一种）

  • git remote -v 查看远程仓库地址
  • git remote add origin 远程仓库地址 添加远程仓库，origin是远程仓库别名
  • git remote rm origin 移除远程仓库origin
  1. 本地仓库推送远程仓库
  • git push 远程仓库 远程仓库分支 如： git push origin master
  1. 推送远程仓库 简写版
  • git push -u origin master 记住当前推送的仓库和分支
  • git push 下次就直接推送
  1. 克隆远程仓库
  • git clone 远程仓库地址
  1. 同步远程代码
  • git pull
  1. 版本回退
  • 本地历史库回退 git reset --hard 版本号
  • 强制推到远程 git push -f

git解决冲突 ​

出现在合并代码时（主要是修改同个文件）：在下拉代码（就是合并代码）的时候远程代码会和本地仓库代码出现冲突；在合并分支时候两个分支代码冲突。

方法一：下拉的时候合并代码，再提交（vscode自带对比工具）

  • Accpet Current Change 接收自己的
  • Accpet Incoming Change 接收远程的
  • Accpet Both Changes 都接收
  • Compare Changes 自己对比之后报存

方法二：但是 git pull 相对不安全，因为会自动和本地内容合并，我们也可以选择使用 git fetch

shell
# 使用 fetch 获取远程master最新信息并开辟一个临时分支tmp
$ git fetch origin master:tmp

# 讲当前分支和临时分支的内容进行对比
$ git diff tmp

# 再选择合并分支内容
$ git merge tmp
